description: "Reference for Breadbox rules endpoints: list, get, create, update, delete rules, apply retroactively, and dry-run conditions against existing transactions."
---

Rules automatically categorize transactions during sync by matching conditions on transaction fields. Each rule contains a condition tree that Breadbox evaluates against every new or modified transaction. When the condition matches, the rule applies its configured action — typically setting the transaction's category. Rules run in priority order (lower number = runs first) according to a four-stage pipeline.

---

## List rules

```text
GET /api/v1/rules
```

Returns transaction rules with cursor pagination, ordered by creation time descending.

**Auth:** Read

### Query parameters

<ParamField query="enabled" type="boolean">
  Filter to only enabled (`true`) or only disabled (`false`) rules. Omit to return both.
</ParamField>

<ParamField query="category_slug" type="string">
  Filter to rules that set the given category (via the `set_category` action).
</ParamField>

<ParamField query="search" type="string">
  Case-insensitive substring match against rule name.
</ParamField>

<ParamField query="cursor" type="string">
  Opaque pagination cursor from a previous response's `next_cursor`. Omit to fetch the first page.
</ParamField>

<ParamField query="limit" type="integer" default="50">
  Results per page. Minimum `1`, maximum `500`.
</ParamField>

### Response

```json
{
  "rules": [
    {
      "id": "r1a2b3c4-d5e6-f789-0abc-def123456789",
      "short_id": "Pq7rSt8u",
      "name": "Amazon purchases",
      "enabled": true,
      "priority": 10,
      "trigger": "on_create",
      "conditions": {
        "and": [
          { "field": "name", "op": "contains", "value": "AMAZON" }
        ]
      },
      "actions": [
        { "type": "set_category", "category_slug": "shopping-online" }
      ],
      "category_id": "cat_abc12345",
      "category_slug": "shopping-online",
      "category_display_name": "Online shopping",
      "created_by_type": "user",
      "created_by_id": "u4Xm9pQ2",
      "created_by_name": "Ricardo",
      "hit_count": 42,
      "last_hit_at": "2026-04-23T12:00:00Z",
      "created_at": "2025-12-01T10:00:00Z",
      "updated_at": "2025-12-01T10:00:00Z"
    }
  ],
  "next_cursor": "",
  "has_more": false,
  "total": 1
}
```

The `category_*` fields are derived from the rule's first `set_category` action and are omitted when no such action exists. Pipeline `stage` is not surfaced on response — use the numeric `priority` (baseline=0, standard=10, refinement=50, override=100).

```bash
curl -H "X-API-Key: bb_your_key" \
  "http://localhost:8080/api/v1/rules?enabled=true&limit=100"
```

## Create a rule

```text
POST /api/v1/rules
```

Creates a new categorization rule. The rule runs automatically on every future sync. To apply it to existing transactions immediately, use `POST /api/v1/rules/{id}/apply` after creation.

**Auth:** Write

### Request body

```json
{
  "name": "Netflix subscription",
  "stage": "standard",
  "trigger": "on_create",
  "conditions": {
    "and": [
      { "field": "name", "op": "contains", "value": "NETFLIX" },
      { "field": "amount", "op": "gt", "value": 0 }
    ]
  },
  "category_slug": "entertainment-streaming-services"
}
```

<ParamField body="name" type="string" required>
  Human-readable name for the rule (e.g., `"Netflix subscription"`).
</ParamField>

<ParamField body="stage" type="string" default="standard">
  Pipeline stage. One of: `baseline`, `standard`, `refinement`, `override`. Resolves to priority `0`, `10`, `50`, or `100` respectively. Stage names are case-insensitive. If both `stage` and `priority` are provided, `priority` wins.
</ParamField>

<ParamField body="priority" type="integer">
  Raw pipeline priority integer. Lower numbers run first. Overrides `stage` when both are provided.
</ParamField>

<ParamField body="trigger" type="string" default="on_create">
  When the rule fires. One of: `on_create` (default), `on_change`, `always`. `on_update` is accepted as an alias for `on_change`.
</ParamField>

<ParamField body="conditions" type="object" required>
  The condition tree to evaluate. See [Condition structure](#condition-structure) below. A zero-value `{}` matches every transaction.
</ParamField>

<ParamField body="category_slug" type="string">
  Sugar for a single `set_category` action. Provide this **or** an explicit `actions` array — one of the two is required.
</ParamField>

<ParamField body="actions" type="array">
  Explicit list of action objects to apply when the conditions match. Each action has a `type` (e.g., `set_category`, `add_tag`, `add_comment`) and type-specific fields like `category_slug`, `tag_slug`, or `content`.
</ParamField>

<ParamField body="expires_in" type="string">
  Optional Go-format duration (e.g., `"720h"`) after which the rule auto-disables.
</ParamField>

### Response

Returns the created rule with `201 Created`.

```bash
curl -X POST \
  -H "X-API-Key: bb_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Netflix subscription",
    "stage": "standard",
    "conditions": {
      "field": "name",
      "op": "contains",
      "value": "NETFLIX"
    },
    "category_slug": "entertainment-streaming-services"
  }' \
  "http://localhost:8080/api/v1/rules"
```

## Update a rule

```text
PUT /api/v1/rules/{id}
```

Updates an existing rule. Every field is optional — only the keys you include in the body are changed. Omitted fields keep their current values.

**Auth:** Write

### Path parameters

<ParamField path="id" type="string" required>
  The rule's Breadbox UUID or short ID.
</ParamField>

### Request body

<ParamField body="name" type="string">Rename the rule.</ParamField>
<ParamField body="conditions" type="object">Replace the condition tree. Same shape as create.</ParamField>
<ParamField body="actions" type="array">Replace the action list.</ParamField>
<ParamField body="category_slug" type="string">Sugar for replacing the rule's actions with a single `set_category`.</ParamField>
<ParamField body="trigger" type="string">`on_create`, `on_change`, or `always`.</ParamField>
<ParamField body="stage" type="string">`baseline`, `standard`, `refinement`, or `override`. If both `stage` and `priority` are sent, `priority` wins.</ParamField>
<ParamField body="priority" type="integer">Raw pipeline priority. Lower runs first.</ParamField>
<ParamField body="enabled" type="boolean">Toggle the rule on or off without deleting it.</ParamField>
<ParamField body="expires_at" type="string (ISO 8601)">Set or clear the auto-disable timestamp. Pass an empty string to clear.</ParamField>

### Response

Returns the updated rule object (same shape as List → `rules[i]`).

```bash
curl -X PUT \
  -H "X-API-Key: bb_your_key" \
  -H "Content-Type: application/json" \
  -d '{"enabled": false}' \
  "http://localhost:8080/api/v1/rules/r1a2b3c4-d5e6-f789-0abc-def123456789"
```

## Delete a rule

```text
DELETE /api/v1/rules/{id}
```

### Path parameters

<ParamField path="id" type="string" required>

</ParamField>

```bash
  -H "X-API-Key: bb_your_key" \
```

## Apply a rule retroactively

```text
POST /api/v1/rules/{id}/apply
```

Runs a single rule against all existing transactions in the database. Transactions that already have a manual `category_override` are skipped. Returns the number of transactions updated.

**Auth:** Write

### Path parameters

<ParamField path="id" type="string" required>
  The rule's Breadbox UUID or short ID.
</ParamField>

### Response

```json
{
  "rule_id": "r1a2b3c4-d5e6-f789-0abc-def123456789",
  "affected_count": 23
}
```

```bash
curl -X POST \
  -H "X-API-Key: bb_your_key" \
  "http://localhost:8080/api/v1/rules/r1a2b3c4-d5e6-f789-0abc-def123456789/apply"
```

## Apply all rules retroactively

```text
POST /api/v1/rules/apply-all
```

Runs all active rules against all existing transactions, in priority order. Transactions with a `category_override` are skipped. This is equivalent to re-running the full categorization pipeline on historical data.

**Auth:** Write

### Response

The `rules_applied` object maps each applied rule's UUID to the number of transactions it affected. `total_affected` is the sum across all rules.

```json
{
  "rules_applied": {
    "r1a2b3c4-d5e6-f789-0abc-def123456789": 34,
    "r2b3c4d5-e6f7-890a-bcde-f12345678901": 108
  },
  "total_affected": 142
}
```

```bash
curl -X POST \
  -H "X-API-Key: bb_your_key" \
  "http://localhost:8080/api/v1/rules/apply-all"
```

## Preview a condition (dry run)

```text
POST /api/v1/rules/preview
```

Runs a condition tree against existing transactions without making any changes. Returns the matching transactions so you can verify a condition before saving it as a rule.

**Auth:** Write

### Request body

```json
{
  "conditions": {
    "and": [
      { "field": "name", "op": "contains", "value": "AMAZON" },
      { "field": "amount", "op": "gt", "value": 50 }
    ]
  },
  "sample_size": 20
}
```

<ParamField body="conditions" type="object" required>
  The condition tree to evaluate. Uses the same structure as rule conditions.
</ParamField>

<ParamField body="sample_size" type="integer" default="50">
  Maximum number of matching transactions to return in the preview sample.
</ParamField>

### Response

```json
{
  "match_count": 34,
  "sample": [
    {
      "id": "t1a2b3c4-d5e6-f789-0abc-def123456789",
      "name": "AMAZON.COM*AB1CD2EF3",
      "amount": 89.99,
      "date": "2026-01-15"
    }
  ]
}
```

```bash
curl -X POST \
  -H "X-API-Key: bb_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "conditions": {
      "and": [
        { "field": "name", "op": "contains", "value": "AMAZON" }
      ]
    }
  }' \
  "http://localhost:8080/api/v1/rules/preview"
```

## Condition structure

Rules use a recursive JSON condition tree. A condition node is either a **leaf** (`field` / `op` / `value`) or a **combinator** (`and`, `or`, or `not` nesting other conditions). Mixing a leaf and a combinator in the same node is rejected. A zero-value `{}` matches every transaction. See the canonical [rule DSL spec](https://github.com/canalesb93/breadbox/blob/main/docs/rule-dsl.md) for the full grammar.

### Combinators

| Combinator | Description |
| --- | --- |
| `and` | Array of sub-conditions that must **all** match. |
| `or`  | Array of sub-conditions where **any** may match. |
| `not` | A single sub-condition that must **not** match. |

```json
{
  "and": [
    { "field": "name", "op": "contains", "value": "AMAZON" },
    {
      "or": [
        { "field": "amount", "op": "gt", "value": 50 },
        { "field": "merchant_name", "op": "eq", "value": "Amazon" }
      ]
    }
  ]
}
```

### Leaf nodes

| Field | Description |
| --- | --- |
| `field` | Transaction field to evaluate (see available fields below). |
| `op` | Comparison operator (see operators below). |
| `value` | The value to compare against. |

### Available fields

| Field | Type | Description |
| --- | --- | --- |
| `name` | string | Raw transaction name from the institution. |
| `merchant_name` | string | Cleaned merchant name from the provider. |
| `amount` | numeric | Transaction amount (Plaid convention: positive = debit). |
| `category_primary` | string | Raw provider primary category. |
| `category_detailed` | string | Raw provider detailed category. |
| `category` | string | Assigned Breadbox category slug. |
| `pending` | boolean | Whether the transaction is pending. |
| `provider` | string | Data provider (`plaid`, `teller`, `csv`). |
| `account_id` | string | Breadbox account UUID. |
| `account_name` | string | Account display name. |

### Operators by field type

**String operators**

| Operator | Description |
| --- | --- |
| `eq` | Exact match (case-sensitive). |
| `neq` | Not equal. |
| `contains` | Case-insensitive substring match. |
| `not_contains` | Does not contain the substring. |
| `matches` | Regular expression match (RE2, case-sensitive; prefix with `(?i)` for insensitive). |
| `in` | Value is one of a list (pass `value` as an array). |

**Numeric operators**

| Operator | Description |
| --- | --- |
| `eq` | Equal. |
| `neq` | Not equal. |
| `gt` | Greater than. |
| `gte` | Greater than or equal. |
| `lt` | Less than. |
| `lte` | Less than or equal. |

**Boolean operators**

| Operator | Description |
| --- | --- |
| `eq` | Equal to `true` or `false`. |
| `neq` | Not equal. |

## Pipeline stages

Rules run in priority order during sync. Lower priority numbers run first, so earlier stages can set a baseline that later stages can refine or override.

| Stage | Priority | Description |
| --- | --- | --- |
| `baseline` | `0` | Broad catch-all rules that categorize the most common transactions. |
| `standard` | `10` | Default stage. Regular categorization rules. |
| `refinement` | `50` | Rules that correct or narrow categories set by earlier stages. |
| `override` | `100` | High-confidence rules that should always win, regardless of earlier results. |

Transactions with a manual `category_override` are skipped by all pipeline stages.

## Error codes

| Condition | Status | Code |
| --- | --- | --- |
| Missing or invalid API key | `401` | `MISSING_API_KEY` / `INVALID_API_KEY` |
| Rule not found | `404` | `NOT_FOUND` |
| Missing `name` or no action provided | `400` | `VALIDATION_ERROR` |
| Unknown `stage` value or invalid condition | `400` | `VALIDATION_ERROR` |
| Invalid `cursor` | `400` | `INVALID_PARAMETER` |
| Referenced `category_slug` does not exist | `400` | `VALIDATION_ERROR` |
