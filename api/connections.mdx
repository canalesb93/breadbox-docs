Connections represent the links between Breadbox and your financial institutions. Each connection corresponds to one institution login and may contain multiple accounts. The connections API lets you inspect connection health, check recent sync activity, and trigger manual syncs on demand.

---

## List connections

```text
GET /api/v1/connections
```

Returns all bank connections in the order they were created.

**Auth:** Read

### Query parameters

<ParamField query="user_id" type="string (UUID)">
  Filter to connections belonging to a specific family member. If omitted, all connections are returned.
</ParamField>

### Connection status values

| Value | Description |
| --- | --- |
| `active` | Connection is healthy and syncing normally. Transactions are being imported on schedule. |
| `error` | The most recent sync encountered an error. Check `error_code` and `error_message` for details. Breadbox will retry on the next scheduled sync. |
| `pending_reauth` | The institution requires the user to re-authenticate. This is typically triggered by a provider error such as `ITEM_LOGIN_REQUIRED`. The user must complete re-authentication from the admin dashboard before syncing can resume. |
| `disconnected` | The connection has been manually disconnected and is no longer active. Accounts and historical transactions are preserved. |

```bash
curl -H "X-API-Key: bb_your_key" \
  "http://localhost:8080/api/v1/connections"
```

## Get connection status

```text
GET /api/v1/connections/{id}/status
```

Returns detailed health and sync information for a single connection, including the most recent sync log entry.

**Auth:** Read

### Path parameters

<ParamField path="id" type="string" required>
  The connection's Breadbox UUID or short ID.
</ParamField>

### Response

```json
  "short_id": "Cd4eFg5h",
  "user_id": "u1a2b3c4-d5e6-f789-0abc-def123456789",
  "user_name": "Alex Canales",
  "provider": "plaid",
  "institution_id": "ins_3",
  "institution_name": "Chase",
  "status": "active",

  "created_at": "2025-11-01T09:00:00Z",
  "updated_at": "2026-02-28T14:30:00Z",
  "last_sync_log": {
    "id": "s1a2b3c4-d5e6-f789-0abc-def123456789",

```

## Trigger a manual sync

```text
POST /api/v1/sync
```

Triggers a manual data sync. The sync runs asynchronously in the background — this endpoint returns immediately with `202 Accepted` and does not wait for completion. Use `GET /api/v1/connections/{id}/status` to poll for results.

**Auth:** Write

### Request body

The request body is optional. If provided, it must be valid JSON.

```json
{
  "connection_id": "c9d8e7f6-a5b4-3210-fedc-ba0987654321"
}
```

<ParamField body="connection_id" type="string (UUID)">
  The specific connection to sync. If omitted, all active connections are synced sequentially.
</ParamField>

### Response — 202 Accepted

```json
{
  "status": "sync_triggered"
}
```

The same response is returned whether you target a specific `connection_id` or omit the body to sync every active connection. Poll `GET /api/v1/connections/{id}/status` to observe progress.

<CodeGroup>

```bash Sync a specific connection
curl -X POST \
  -H "X-API-Key: bb_your_key" \
  -H "Content-Type: application/json" \
  -d '{"connection_id": "c9d8e7f6-a5b4-3210-fedc-ba0987654321"}' \
  "http://localhost:8080/api/v1/sync"
```

```bash Sync all connections
curl -X POST \
  -H "X-API-Key: bb_your_key" \
  "http://localhost:8080/api/v1/sync"
```

</CodeGroup>

## Connection response fields

<ResponseField name="id" type="string (UUID)" required>
  Breadbox-assigned UUID for the connection.
</ResponseField>

<ResponseField name="institution_id" type="string">
  Provider-assigned institution identifier (e.g., Plaid's `ins_3` for Chase).
</ResponseField>

<ResponseField name="institution_name" type="string" required>
  Human-readable institution name (e.g., `"Chase"`).
</ResponseField>

<ResponseField name="provider" type="string" required>
  Data provider for this connection. One of: `plaid`, `teller`, `csv`.
</ResponseField>

<ResponseField name="status" type="string" required>
  Connection health status. One of: `active`, `error`, `pending_reauth`, `disconnected`.
</ResponseField>

<ResponseField name="error_code" type="string">
  Machine-readable error code from the last failed sync (e.g., `"ITEM_LOGIN_REQUIRED"`). `null` when the connection is healthy.
</ResponseField>

<ResponseField name="error_message" type="string">
  Human-readable description of the error. `null` when the connection is healthy.
</ResponseField>

<ResponseField name="last_synced_at" type="string (ISO 8601)">
  Timestamp of the most recently completed successful sync. `null` if the connection has never synced successfully.
</ResponseField>

<ResponseField name="last_attempted_sync_at" type="string (ISO 8601)">
  Timestamp of the most recently attempted sync, regardless of outcome. `null` if never attempted.
</ResponseField>

<ResponseField name="user_id" type="string (UUID)">
  UUID of the family member who owns this connection.
</ResponseField>

<ResponseField name="user_name" type="string">
  Display name of the owning family member.
</ResponseField>

<ResponseField name="last_sync_log" type="object">
  The most recent sync log entry (only returned by `GET /connections/{id}/status`). `null` if no sync has ever run.

  <Expandable title="last_sync_log fields">
    <ResponseField name="id" type="string (UUID)" required>
      Sync log entry UUID.
    </ResponseField>
    <ResponseField name="short_id" type="string" required>
      8-character base62 alias.
    </ResponseField>
    <ResponseField name="connection_id" type="string (UUID)" required>
      UUID of the parent connection.
    </ResponseField>
    <ResponseField name="trigger" type="string" required>
      What triggered the sync: `cron`, `webhook`, `manual`, or `initial`.
    </ResponseField>
    <ResponseField name="status" type="string" required>
      Sync status: `in_progress`, `success`, or `error`.
    </ResponseField>
    <ResponseField name="added_count" type="integer" required>
      Number of new transactions imported.
    </ResponseField>
    <ResponseField name="modified_count" type="integer" required>
      Number of existing transactions updated.
    </ResponseField>
    <ResponseField name="removed_count" type="integer" required>
      Number of transactions soft-deleted.
    </ResponseField>
    <ResponseField name="error_message" type="string">
      Error message when `status` is `error`. `null` on success.
    </ResponseField>
    <ResponseField name="started_at" type="string (ISO 8601)">
      When the sync began.
    </ResponseField>
    <ResponseField name="completed_at" type="string (ISO 8601)">
      When the sync completed. `null` if still in progress.
    </ResponseField>
    <ResponseField name="duration" type="string">
      Human-formatted duration (e.g., `"15s"`, `"2m4s"`). Omitted when the sync is still in progress.
    </ResponseField>
    <ResponseField name="duration_ms" type="integer">
      Same elapsed time in milliseconds. Convenient for charting or thresholding without parsing the human-formatted `duration`. Omitted when the sync is still in progress.
    </ResponseField>
  </Expandable>
</ResponseField>

## Error codes

| Condition | Status | Code |
| --- | --- | --- |
| Missing or invalid API key | `401` | `MISSING_API_KEY` / `INVALID_API_KEY` |
| Connection not found | `404` | `NOT_FOUND` |
| Internal sync failure | `500` | `INTERNAL_ERROR` |

The `user_id` and `connection_id` parameters are passed through to the service layer without strict UUID-shape validation — short IDs are accepted, and unknown values resolve to a `404 NOT_FOUND` rather than a `400`.
